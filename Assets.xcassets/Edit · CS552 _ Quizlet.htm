Framework
A set of cooperating classes that represent reusable designs of software systems in a particular application domain. More specifically, a framework consists of a set of abstract classes and interfaces which are parts of a semi-complete application that can be specialized to build custom applications..

An application framework consists of a set of abstract classes and interfaces to be specialized. Its changeable aspects are referred to as hot spots, typically hook methods. Custom application make use of the framework by extending and or implementing the classes and interfaces in the framework and overriding the hook methods to provide custom behavior.

Singleton Design Pattern
Intent: Guarantees single instance of a class and provides global access. Applicability:Use this when there must be exactly one instance of a class and it must also be accessible from a well-known point. Structure:Singleton- static getInstance() operation() getData()- static Singleton theInstance data Participant roles: Singleton declare the unique instance of the class as a static variable, and defines the static method getInstance() for client access.

Template Method
Intent: To define the skeleton of an algorithm in a method. Deferring some steps to subclasses. Applicability: Use this when refactoring common behavior among subclasses to avoid code duplication. Structure: GenericClass- templateMethod() hookMethod() ConcreteClass- hookMethod() Participant roles: GenericClass defines the skeleton algorithm templateMethod() along with the abstract hookMethod(). Concrete class is a subclass of GenericClass implementing hookMethod().

Strategy Pattern
Intent: To define a family of algorithms, encapsulate each one, and make them interchangeable. Applicability: Use this when different variants of an algorithm are needed. Structure: Context- ContextMethod() Strategy- algorithm() ConcreteStrategyA- algorithm() Participant roles: Strategy declares an interface common to all supported algorithms. ConcreteStrategy implements the algorithm in the Strategy interface. Context maintains references to one or more Strategy objects.

Iterator Pattern
Intent: To provide a way to access elements of a collection sequentially. Applicability: Use this when you want to access the contents of a collection without exposing it internal representation. Structure: AbstractCollection- iterator() ConcreteCollection- iterator() Iterator- hasNext() next() ConcreteIterator- hasNext() next() Participant roles: Iterator defines an interface for access and traversal of elements. ConcreteIterator implements Iterator and keeps track of the current position. AbstractCollection defines a method for creating an iterator. ConcreteCollection overrides the iterator method to return an instance.
